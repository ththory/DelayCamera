# PRD — Delay Camera (운동 자세 점검용 지연 카메라)

## 1. 배경·목적

### 1.1 배경
운동·스포츠 동작 학습에서 "방금 한 동작을 본인이 직접 본다"는 것은 자세 교정에 매우 효과적이다.
일반 카메라는 영상을 저장 후 되돌려 봐야 해서 흐름이 끊기고, 시중 "딜레이 미러" 앱은 광고·결제·네트워크 전송 등이 섞여 있어 헬스장·체육관에서 신뢰 있게 사용하기 어렵다.

### 1.2 목적
- N초(사용자 조절) 후 자신의 모습을 거울처럼 즉시 볼 수 있는 **저지연 온디바이스 딜레이 미러** 제공.
- AI 분석 일체 없음 — "보여주기"만에 집중하여 단순·신뢰·고성능을 확보한다.
- 백엔드·네트워크 일체 없음 — 헬스장/체육관 Wi-Fi 환경 제약, 프라이버시 우려에서 자유.

### 1.3 재생 시맨틱 (중요)
**연속 지연 재생 (Continuous Delayed Playback)** — "스냅샷 되감기"가 아닌 "지속 재생" 방식.
- 예: 딜레이 = 15초 설정 시
  - 0~15초 (워밍업): 화면은 빈 상태 / 카운트다운 / 라이브 미리보기 중 택1 (UX 후속 결정)
  - 16초 시점: 캡처 1초의 프레임이 재생됨
  - 17초 시점: 캡처 2초의 프레임이 재생됨
  - … 이후 매 프레임이 정확히 15초 늦게 연속 재생됨 (라이브와 동일한 fps)
- 사용자가 슬라이더로 딜레이를 조절하면 버퍼 길이가 변경되며 재생 시점이 부드럽게 이동.

**리플레이 모드 (Replay)**
- 사용자가 "리플레이" 버튼을 누르면 **현재 딜레이 설정 길이(N초)** 만큼의 최근 스트림을 처음부터 1회 재생.
- 재생 중에도 라이브 캡처와 버퍼링은 계속 진행 (버퍼 손실 없음).
- 재생 중 "저장" 버튼으로 해당 N초 클립을 MP4로 저장 가능.
- 재생 완료 후 자동으로 연속 지연 재생 모드로 복귀.

### 1.4 비목표 (Non-Goals)
- 자동 자세 분석·코칭 메시지 (AI 미사용)
- 클라우드 저장·공유 기능
- 영상 편집·후처리
- iOS·Web 지원

## 4. 비기능 요구사항 (NFR)

> **기준 기기**: Samsung Galaxy S26 Ultra (개발·성능 측정 baseline). 모든 정량 NFR은 이 기기에서 측정한다.
> 보조 측정 기기: 1세대 전 플래그십 (S25 Ultra 또는 동급) — 폴백 시나리오 검증용.

| ID | 요구사항 | 측정 기준 (Galaxy S26 Ultra 기준) |
|---|---|---|
| NFR-1 | 저지연 라이브 | 캡처→화면 motion-to-photon ≤ **60ms** (1080p/60fps) |

| NFR-8 | 배터리 소모 | 30분 사용 시 배터리 ≤ **10%** 감소 (디스플레이 밝기 50%, 1080p@60fps) |

| NFR-11 | 저장 처리량 | FR-7 녹화 시 1080p@60fps를 실시간 인코딩, 저장 실패율 ≤ 0.1% |
| NFR-12 | 네트워크 미사용 | `INTERNET` 권한 미선언, lint·런타임 양쪽에서 검증 |

## 5. 핵심 사용 시나리오 (Happy Path)

```
1. 사용자가 앱을 실행한다 → 라이브 프리뷰 ≤ 1.0s
2. 카메라 권한을 허용한다
3. 슬라이더로 15초 딜레이를 설정한다
4. 0~15초: 워밍업 (라이브 미니뷰 + 카운트다운 오버레이)
5. 16초~: 캡처 1초 프레임이 화면에 재생됨. 이후 매 프레임이 정확히 15초 지연으로 연속 재생
6. 사용자는 운동을 계속 수행하면서 15초 전 자기 동작을 동시에 시청
7. 슬라이더로 5초로 변경 → 버퍼가 적응적으로 축소되며 매끄럽게 5초 딜레이로 전환
8. 방금 한 동작을 다시 보고 싶어 "리플레이" 버튼을 누른다 → 직전 5초가 처음부터 재생됨
9. 리플레이 재생 중 "저장" 버튼을 눌러 해당 5초 클립을 갤러리에 MP4로 저장
10. 리플레이 종료 후 자동으로 연속 5초 딜레이 재생 모드로 복귀, 라이브 버퍼는 끊김 없음
11. 발열 경고 알림 발생 (THERMAL_STATUS_MODERATE) → 사용자가 "1080p@30fps로 다운스케일" 선택
12. 앱을 종료 → 미저장 영상은 모두 메모리에서 폐기
```

## 6. 제약·가정

### 6.1 기술 제약
- Kotlin 2.1+, Jetpack Compose, CameraX(+ Camera2 Interop), OpenGL ES 3.0 (CLAUDE.md 합의)
- 인코더: `MediaCodec` HW 인코더 (H.264 Baseline/Main, HEVC Main10 — S26 Ultra HW 지원 활용)
- 네이티브 의존성 최소화 (FFmpeg 등 대용량 라이브러리 회피 목표)
- `INTERNET` permission 절대 선언 금지

### 6.2 기준 기기 (Galaxy S26 Ultra) 가정
- SoC: 최신 플래그십 (Snapdragon 8 Gen 5 또는 Exynos 2500급) — 1080p@60fps 인코딩 여유
- RAM: 12~16GB → 60초 딜레이 시 2.5GB 메모리 사용 여유
- 디스플레이: 120Hz 이상 → 60fps 출력 매끄럽게 표시
- 카메라: 후면 메인 ≥ 50MP, 4K 60fps 캡처 지원 (YUV_420_888 / Surface)
- GPU: OpenGL ES 3.2 / Vulkan 1.3 — OpenGL ES 3.0만 사용 (이식성)
- 저장공간: UFS 4.0급 → 1080p@60fps 실시간 저장 처리량 충분

### 6.3 가정
- 대상 기기 하한: Android 8.0 이상, RAM ≥ 6GB, GPU OpenGL ES 3.0 지원
- 카메라 출력: YUV_420_888 또는 RGBA Surface
- 단일 사용자, 가로/세로 단일 화면 (멀티 윈도우는 P2)

### 6.4 의존
- 후속 ADR(workflow-create-architecture)이 다음을 결정해야 함:
  - 프레임 버퍼 저장 형태 (raw YUV 풀 vs 압축 인코딩 큐 vs GPU 텍스처 풀)
  - 최대 지원 딜레이 시간 (메모리·발열·UX 트레이드오프, 잠정 60초)
  - 렌더 파이프라인 (Surface ↔ OpenGL ES 통신 방식, SurfaceTexture vs ImageReader)
  - 저장 파이프라인 (실시간 인코딩 vs 버퍼 덤프, MediaMuxer 통합 방식)
  - 발열 대응 정책 (자동 다운스케일 단계, A-5 해소)
